# 왜 SW공학이 다시 주목받는가

## 과거와 현재의 차이

**과거:** "어떻게 코딩할 것인가?"가 화두

**현재:** "AI가 만든 결과물을 어떻게 검증하고, 연결하며, 안전하게 운영할 것인가?"

AI는 개발의 **엔진**이 되고, 소프트웨어 공학은 그 엔진을 제어하는 **조향 장치와 브레이크** 역할을 수행합니다.

## 구현 속도가 병목이 아니다

예전에는 **구현 속도**가 병목이었습니다. 이제는 **무엇을 어떻게 만들어야 하는지 결정하는 능력**이 병목이 됩니다.

AI는 함수, 클래스, CRUD, 테스트 초안을 빠르게 만들 수 있지만, 다음 문제들은 여전히 **시스템적 판단**이 필요합니다.

- 서비스 경계 설정
- 데이터 일관성 유지
- 장애 격리 전략
- 보안 모델 설계
- 성능 예산 배분
- 진화 가능한 구조 설계

따라서 AI 시대에는 코딩보다 **아키텍처, 품질속성, 트레이드오프 판단**이 더 비싸고 더 중요한 역량이 됩니다.

## DORA 2025의 진단

DORA는 2025 보고서에서 AI의 핵심 역할을 이렇게 정의합니다.

> "AI는 조직의 기존 강점과 약점을 **증폭**하는 것이다."

공학적 기반이 약하면 AI는 생산성보다 **부채를 더 빨리** 키울 수 있습니다. 반대로 공학 기반이 탄탄한 조직은 AI를 **진정한 증폭기**로 활용합니다.

## 실무적 해석

팀 차원에서 질문이 바뀌어야 합니다.

> "AI를 쓸까 말까?" (X)

> "AI가 빨라진 만큼, 우리의 설계·리뷰·문서·테스트 체계도 같이 강화됐는가?" (O)

그렇지 않으면 몇 달 뒤에 다음 문제들이 쌓입니다.

- 중복 코드
- 리뷰 불능 PR
- 약한 테스트
- 설명 불가능한 구조
- 숨은 기술 부채

반대로 소프트웨어 공학이 살아 있는 팀은 AI를 단순 **코드 생성기**가 아니라 **설계 의도를 구현하는 증폭기**로 사용할 수 있습니다.
