# 관용적인 수신과 엄격한 송신, 포스텔의 법칙 (Postel's Law)

## I. 견고한 시스템 연동의 대원칙, 포스텔의 법칙 개요


**정의** : "자신이 행하는 일은 엄격하게, 남으로부터 받아들이는 일은 관대하게 하라(Be conservative in what you do, be liberal in what you accept from others)"는 원칙으로, **견고성 원칙**(Robustness Principle)이라고도 함  

**핵심 특징 및 시사점** :  
( **상호 운용성 보장** ) 다양한 환경의 구현체들이 서로 통신할 때 발생하는 미세한 차이를 수용하여 전체 네트워크의 연결성 유지  
( **견고한 인터페이스** ) 존 포스텔( **Jon Postel** )이 **TCP** 명세(RFC 793)에서 제안했으며, 분산 시스템의 안정적 결합을 위한 필수 지침  
( **송신 시 엄격성** ) 내가 보내는 데이터는 표준 명세를 철저히 준수하여 상대방이 처리하는 데 혼선이 없도록 보장  
( **수신 시 관용성** ) 상대방이 보낸 데이터에 약간의 오류나 비표준적 요소가 있더라도, 해석이 가능하다면 서비스를 중단하지 않고 처리

### 나. 시스템 설계 시 포스텔의 법칙 적용 가이드

| 적용 구분 | 주요 행동 지침 | 보안 및 운영적 가치 |
|:---:|--------------|------------------|
| **송신 (Conservative)** | 표준 규격을 100% 준수하고 모호한 데이터 전송 금지 | 상대방 시스템의 에러 발생 가능성 원천 차단 |
| **수신 (Liberal)** | 필수 데이터가 존재한다면 부가적 형식 오류는 무시 | 서비스 가용성( **Availability** ) 및 호환성 향상 |
| **에러 핸들링** | 잘못된 수신 데이터라도 즉각 중단보다는 보완 처리 시도 | 비즈니스 연속성 유지 및 사용자 경험 개선 |
| **로그 기록** | 관용적으로 처리하되, 표준 위반 사항은 반드시 로그 기록 | 잠재적 오작동 원인 분석 및 상대방 개선 유도 |

## III. 포스텔의 법칙과 보안 관리의 딜레마

### 가. 관용적 수신(Liberal)의 보안적 위험 요소

포스텔의 법칙은 상호 운용성에는 탁월하지만, 보안 측면에서는 **공격 표면**을 넓히는 부작용이 있을 수 있습니다.

| 보안 취약 요인 | 상세 내용 | 대응 전략 |
|:---:|----------|----------|
| **입력값 조작** | 관용적인 파싱 로직을 악용한 페이로드 삽입 | 화이트리스트 기반의 입력값 검증 병행 |
| **HTTP Smuggling** | 요청의 모호함을 이용한 캐시 오염 및 권한 우회 | 보안 장비와 서버 간 파싱 규격 통일 |
| **회피 공격 (Evasion)** | 보안 장비가 관용적으로 무시하는 부분을 이용한 침투 | 엄격한 시그니처 매칭 및 프로토콜 검사 |

### 나. 실무적 균형 전략: 안전한 포스텔의 법칙 적용
- **보안 임계치 설정** : 의미 전달에 영향이 없는 형식적 오류(예: 불필요한 공백)는 관용하되, 보안과 직결된 데이터(예: SQL 예약어)는 엄격히 차단
- **정규화 (Normalization)** : 수신된 데이터를 관대하게 받아들인 후, 내부 처리 전 반드시 표준 형식으로 정규화하여 가공
- **상호 인증 강화** : 관용적 처리를 하더라도 신뢰할 수 있는 대상( **IdP**, **mTLS** )과의 통신에서만 적용 범위를 넓게 설정

> **핵심** : **포스텔의 법칙**은 분산 시스템의 **생존력**을 키워주지만, 보안 관리자는 **관용**이 **취약점**으로 변질되지 않도록 데이터의 정체성을 엄격히 감시하는 문지기 역할을 수행해야 함
