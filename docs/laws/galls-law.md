# 복잡한 시스템의 유일한 생존법, 갈의 법칙 (Gall's Law)

## I. 시스템 진화와 복잡성 제어의 원칙, 갈의 법칙 개요

```mermaid
%%{init: { 'theme': 'base', 'themeVariables': { 'edgeLabelBackground': '#fff' }}}%%
flowchart LR
    A["처음부터 설계된\n거대하고 복잡한 시스템"] -- "작동성(Working)의 점진적 증명 및\n진화적 확장" --> B["단순한 시스템에서 진화한\n작동하는 복잡한 시스템"]
    style A fill:#f9f9f9,stroke:#333,stroke-width:1px
    style B fill:#e1f5fe,stroke:#01579b,stroke-width:1px
```

**정의** : "작동하는 모든 복잡한 시스템은 예외 없이 작동하는 단순한 시스템에서 진화했다"는 원칙으로, 존 갈( **John Gall** )이 자신의 저서 《시스템 이론(Systemantics)》에서 제안한 시스템 설계의 핵심 법칙  

**핵심 특징 및 시사점** :  
( **진화적 탄생** ) 복잡한 시스템은 처음부터 '설계'되는 것이 아니라, 성공적으로 작동하는 단순한 모델에서 단계적으로 '성장'해야 함  
( **빅뱅 설계의 실패** ) 처음부터 모든 기능을 갖춘 복잡한 시스템을 설계하려 하면 결코 의도대로 작동하지 않으며, 이를 패치로 해결하려 해도 성공할 수 없음  
( **작동성 우선** ) 시스템의 규모보다 중요한 것은 "현재 이 시스템이 실제로 작동하는가"이며, 작동하지 않는 복잡성( **Non-working Complexity** )은 기술 부채일 뿐임  
( **점진적 확장** ) 단순한 핵심 기능을 먼저 구현하고, 이를 바탕으로 환경의 변화에 맞춰 기능을 추가해 나가는 것이 복잡성을 다루는 유일한 방법임

## II. 갈의 법칙의 작동 메커니즘과 시스템 구축 전략

### 가. 성공적인 시스템 구축의 진화 모델

```mermaid
graph LR
    Simple["1. 단순한 시스템\n(핵심 기능 작동)"] --> Validate["2. 검증 및 안정화\n(Working Proof)"]
    Validate --> Evolve["3. 기능 확장\n(Incremental Growth)"]
    Evolve --> Complex["4. 복잡한 시스템\n(안정된 거대 구조)"]
    
    style Simple fill:#e3f2fd,stroke:#1e88e5
    style Complex fill:#e1f5fe,stroke:#01579b
```

### 나. 갈의 법칙을 활용한 소프트웨어 엔지니어링 전략

| 전략 항목 | 상세 내용 | 기대 효과 |
|:---:|----------|----------|
| **MVP 개발** | **Minimum Viable Product** (최소 기능 제품)로 시장과 기술성을 먼저 검증 | 초기 투자 리스크 감소 및 작동성 조기 확보 |
| **반복적 개발** | 애자일( **Agile** ) 방법론을 통해 작은 단위의 배포와 피드백 반복 | 시스템의 유연성 및 품질 가시성 확보 |
| **모듈화 설계** | 독립적으로 작동하는 작은 컴포넌트들의 조합으로 시스템 구성 | 부분적 실패가 전체로 확산되는 것 방지 |
| **스트랭글러 패턴** | 레거시( **Legacy** ) 시스템을 점진적으로 새로운 아키텍처로 교체 ( **Strangler Fig** ) | 빅뱅 방식의 시스템 전환 실패 위험 제거 |

## III. 갈의 법칙과 현대적 아키텍처 및 보안의 연계

### 가. 전통적 빅뱅 설계 vs. 현대적 진화형 설계 비교

| 비교 항목 | 빅뱅 설계 (Big Bang Design) | 진화형 설계 (Evolutionary Design) |
|:---:|---------------------------|-------------------------------|
| **설계 철학** | 완벽한 설계 후 구축 ( **Waterfall** ) | 최소 기능에서 점진적 확장 ( **Agile** ) |
| **복잡성 대응** | 처음부터 모든 예외 상황 고려 | 작동하는 단순 모델의 피드백 기반 대응 |
| **장애 대응** | 복잡한 의존성으로 인해 원인 파악 난해 | 계층화된 단순 구조로 문제 격리 용이 |
| **보안 모델** | 전사적 통합 보안 프레임워크 강제 | 서비스별 마이크로 세그멘테이션 및 확장 |

### 나. 보안 시스템 구축에의 적용 제언
- **방어의 단순화** : 복잡하고 난해한 보안 규칙보다는, 명확하고 단순한 차단 규칙에서 시작하여 오탐( **FP** )을 줄이며 고도화해 나갈 것
- **점진적 제로 트러스트** : 전사 인프라를 한꺼번에 제로 트러스트로 바꾸려 하기보다, 특정 중요 어플리케이션부터 단계적으로 적용 범위를 넓힐 것
- **회복 탄력성(Resilience) 중심** : 모든 공격을 막는 완벽한 시스템은 불가능하므로, 단순한 복구 체계에서 시작하여 복잡한 재해 복구( **DR** ) 시스템으로 진화시킬 것

> **핵심** : **갈의 법칙**은 우리에게 "작게 시작하고, 작동하게 만들고, 그 다음에 확장하라"는 단순하지만 강력한 진리를 상기시키며, 이것이 복잡한 소프트웨어 세상을 관리하는 유일한 생존 전략임
