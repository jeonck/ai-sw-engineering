# 공학 기반이 AI 효과를 결정한다

## 좋은 코드 원칙이 더 중요해진 이유

흥미롭게도 AI 시대의 "새로운 비밀"은 완전히 새로운 원칙이 아닙니다. Thoughtworks는 AI-friendly 코드 설계의 핵심으로 다음을 강조합니다.

- **표현력 있는 네이밍**: 변수, 함수, 클래스 이름이 의도를 명확히 드러낼 것
- **모듈성**: 각 모듈이 하나의 책임만 가질 것
- **적절한 추상화**: 복잡성을 숨기되 계약은 명확할 것
- **DRY 원칙**: 중복을 제거해 AI가 일관된 패턴을 학습할 수 있게 할 것

**사람이 읽기 좋은 코드가 AI에게도 읽기 좋고, 변경이 쉬운 구조가 AI에게도 다루기 쉽습니다.**

결국 AI는 소프트웨어 공학을 낡게 만든 것이 아니라, **좋은 공학 원칙의 가치를 더 측정 가능하게** 만든 셈입니다.

## 공학 기반이 약할 때 AI는 무엇을 증폭하는가

| 기반 약점 | AI가 증폭하는 결과 |
|-----------|-------------------|
| 명명 규칙 없음 | AI가 불일치한 이름 대량 생성 |
| 테스트 부재 | 검증 없는 AI 코드 축적 |
| 문서 없음 | AI가 잘못된 맥락으로 코드 생성 |
| 높은 결합도 | AI 수정 시 예상치 못한 side effect |
| 리뷰 문화 없음 | AI 생성 오류가 프로덕션까지 도달 |

## 공학 기반을 강화하는 실천

### 코드 수준

- 의도를 드러내는 함수/변수명 사용
- 단일 책임 원칙 (SRP) 준수
- 테스트 가능한 구조 (의존성 주입, 순수 함수)

### 팀 수준

- PR 크기를 작게 유지 (AI가 생성해도 리뷰 가능한 단위)
- 리뷰 체크리스트 운영
- ADR로 중요 결정 기록

### 조직 수준

- 아키텍처 원칙 문서화
- 코딩 가이드라인 최신화
- 기술 부채 트래킹

## 핵심 메시지

AI는 코드를 더 빠르게 만들어주지만, **무엇을 만들어야 하는지**와 **그것이 올바른지**는 여전히 인간 엔지니어의 판단 영역입니다. 공학적 기반이 탄탄할수록 AI를 더 잘 활용할 수 있습니다.
